Standing directive from Ruan (2026-06-21): every time I build a new feature or add-on for Imara, I must analyse whether cleanup is needed as part of finishing it â€” find dead code, duplicate logic, unused files/components/endpoints, and unnecessary complexity the feature introduced, and remove what isn't necessary. **Why:** Feature-adding is addictive and each one leaves a little mess that piles up; cleanup is where the real work is. Fact-checked and true: AI-assisted coding measurably accumulates cruft (GitClear, 211M lines 2020-24: copy-paste 8.3%â†’12.3%, refactored code 24.1%â†’9.5%, copy-paste first exceeded refactoring in 2024). **How to apply:**
- Use TOOLS, not vibes/LLM opinion: `vulture` (dead code), `ruff --select F401,F811,F841` (unused imports/vars), `ts-prune`/`knip` + eslint no-unused (frontend), `jscpd` (duplication). An LLM asked to "find dead code" will always produce a list, including false positives â€” never delete on its say-so.
- VERIFY every candidate before deleting: grep render-sites / caller-trace, then run the FULL test suite (185+). My own ad-hoc greps false-positived Dashboard/Navbar this session â€” proof you must verify.
- Respect "duplication is cheaper than the wrong abstraction" (Sandi Metz): don't blindly dedup; and PRESERVE intentional defensive redundancy (faithfulness recomputing ratios, deterministic cross-checks) â€” that's a feature, not cruft.
- Prefer a CI ratchet (ruff/vulture/jscpd gates) so cruft can't silently re-accumulate over time.
