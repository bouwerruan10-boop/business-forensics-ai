# Imara — Deep Improvement Plan (research → questions → answers → plan)
*Permission to "over-engineer" — but every item here is chosen because it materially improves the END PRODUCT's accuracy, trust, or durability, not for its own sake.*

## 1. Research synthesis (two rounds)
**LLM/agent evaluation.** Robust evaluation is multi-layer: offline golden tests + human judgment + production telemetry, gated in CI ([Langfuse](https://langfuse.com/blog/2025-10-21-testing-llm-applications), [DeepEval](https://deepeval.com/guides/guides-llm-as-a-judge)). **LLM-as-judge** is only trustworthy if it correlates with humans — validate the judge against a human-labelled golden set and aim for **75–90% agreement before scaling** ([Galileo](https://galileo.ai/blog/llm-as-a-judge-vs-human-evaluation), [Maxim](https://www.getmaxim.ai/articles/building-a-golden-dataset-for-ai-evaluation-a-step-by-step-guide/)). Don't let an LLM write the golden assertions (it locks in current bugs); pin model versions for agent *and* judge.

**RAG / grounding.** Grounding an LLM in authoritative texts is the proven way to cut hallucination and enable **citations** — "particularly crucial in high-stakes domains like law, where factual accuracy is mandatory" ([AWS](https://docs.aws.amazon.com/prescriptive-guidance/latest/agentic-ai-serverless/grounding-and-rag.html), [arXiv legal RAG](https://arxiv.org/pdf/2502.20364)). Lightweight RAG needs no heavy infra: a small curated corpus with keyword/embedding retrieval, or **`sqlite-vec`** (fits Imara's existing SQLite) ([SitePoint](https://www.sitepoint.com/local-first-rag-vector-search-in-sqlite-with-hamming-distance/)). SA legislation is citable by **short title + number + year** (e.g. *VAT Act 89 of 1991*, *Companies Act 71 of 2008*) ([Acts Online](https://www.acts.co.za/)).

**Observability.** Tracing is the key tool — hierarchical traces of every LLM call/tool/retrieval, with cost/latency/quality. **Langfuse** (open-source, OTEL-compatible) is the standard; instrument at agent level, redact PII at the SDK layer; for per-analysis batch jobs 100% sampling is fine ([Langfuse](https://langfuse.com/docs/observability/overview)).

## 2. Questions I asked of the research
- **Q1.** Imara's biggest *credibility* risk is the SA Tax/Legal agents reasoning about specific law from parametric memory — could they cite wrong or outdated provisions? Would grounding them in real SA legislation materially help, and is it feasible in this stack?
- **Q2.** How do we *know* Imara's output quality is good and not regressing? The eval harness only checks deterministic ratios today.
- **Q3.** In production, can we see what each run costs, where time goes, and where hallucinations occur? (No tracing today.)
- **Q4.** Is there low-risk architectural depth (structured outputs, self-critique) that raises quality/cuts cost?

## 3. Answers (from the research)
- **A1 — Yes, and yes.** Grounding is the highest-leverage accuracy/trust move for exactly the most legally-sensitive agents. Feasible without heavy infra: a **curated, citation-tagged corpus of the key SA provisions** Imara actually reasons about (VAT threshold, provisional-tax dates, EMP201, Companies Act director/CIPC duties, BBBEE level thresholds, POPIA principles) + a **deterministic keyword retriever** (no embeddings dependency, fully testable), injected into the SA agents' prompts as authoritative text they must cite. Stable, well-established facts; tag each with an `as_of` date for maintenance. (Embeddings/`sqlite-vec` are a later upgrade.)
- **A2 — Build a real eval framework.** Expand golden cases; add **LLM-as-judge graders** (finding specificity / ZAR-quantification / benchmark-citation / coverage) with a written rubric, validated against a small human-labelled set (target 75–90% agreement), run in the paid `run_evals --full` and gated in CI for the deterministic parts.
- **A3 — Add a tracing seam.** A no-op-by-default tracer that, when `LANGFUSE_*` env is present, records per-call spans (model, latency, cost) across the CEO→waves→agents tree. Lightweight, like the other future-proofing seams.
- **A4 — Yes, incrementally.** Structured outputs can drop the 2nd parse call; a self-critique pass on findings can raise quality. Medium-risk; sequence after grounding + evals so we can measure the change.

## 4. The plan (prioritized for best end product)
1. **★ SA Compliance Knowledge Base + RAG grounding (BUILD NOW).** Curated, dated, citation-tagged corpus of key SA tax + company-law provisions; deterministic keyword retriever; inject relevant provisions (with citations) into the SA Tax + SA Legal agents so they cite *real law* (Act number + section), not memory. Biggest accuracy/trust lever; deterministic + testable. *This cycle.*
2. **Rigorous eval framework** — LLM-as-judge graders + expanded golden set + rubric + CI regression gate. Locks in quality so everything else can improve safely.
3. **Observability/tracing seam** — Langfuse-ready hierarchical tracing + per-run cost/latency. Production maturity + hallucination debugging.
4. **Agent architecture depth** — structured outputs (kill the 2nd call) + a finding self-critique/reflection pass. Measure against the eval framework from (2).
5. **Simulator optimiser** — "best bundle of actions under a budget" (the prescriptive rung), reusing the v2 engine.

Sequencing rationale: grounding (1) is the highest product-quality win and is self-contained; evals (2) make every later change safe and measurable; observability (3) gives the data; (4)/(5) are then measurable, lower-risk increments. All keep Imara's deterministic, citeable, anti-hallucination DNA.

## Implementation log
**Build 1 — SA Compliance RAG grounding: SHIPPED.** `services/sa_knowledge.py` (17 curated, dated, citation-tagged SA tax + company-law provisions; deterministic keyword retriever, no embeddings dependency). `retrieve_grounding(memory, domain)` injects the most relevant provisions — each with its Act + section citation — into the SA Tax and SA Legal agent prompts, so those agents cite **real, dated law** rather than parametric recall. 4 new tests (48 total pass). Embeddings/`sqlite-vec` upgrade deferred until the corpus grows. Next: eval framework (LLM-as-judge), then the observability/tracing seam.

**Build 2 — Quality eval framework: SHIPPED.** `evals/grader.py` gains two **no-API CI regression gates** — `grade_structure` (% of findings meeting the FINDING_RULES contract) and `grade_sa_citation` (% of SA findings citing a specific Act/section — directly measures Build 1 working) — plus `grade_findings_with_judge`, an **LLM-as-judge** semantic grader (rubric: specificity / actionability / grounding / severity_fit, 1–5, pinned `JUDGE_MODEL`, injectable `judge_call`) wired into `run_evals --full`. `evals/RUBRIC.md` documents the rubric and the 75–90% human-alignment validation discipline. 8 new tests across both builds (52 total). Next: observability/tracing seam (Langfuse-ready), then agent-architecture depth.

**Build 3 — Observability/tracing seam: SHIPPED.** `services/tracing.py` adds a per-analysis **token + cost ledger** held in a `contextvars.ContextVar` (correct even under concurrent analyses) plus a no-op-by-default **Langfuse export hook** (lazy import; dormant until `LANGFUSE_*`/`TRACING_ENABLED`). Every Claude call is recorded at the single `_call_claude` chokepoint (real `response.usage` tokens → cost via a per-model price table, env-overridable). `agents/parallel.py` propagates the ledger into worker threads via `contextvars.copy_context()`, so the parallel waves attribute usage correctly. The report gains `llm_usage` (`{calls, input/output tokens, est_cost_usd, by_model}`), surfaced through `GET /api/report/{id}/usage`. 6 new tests (58 total). Frontend cost display + full Langfuse span-tree wiring are the cheap activation follow-ups.

**Build 4 — Deterministic finding self-critique: SHIPPED.** `services/finding_quality.py` grades every finding against Imara's own FINDING_RULES contract and attaches `quality_flags` (`unquantified`, `vague_recommendation`, `no_benchmark`, `no_cost_of_inaction`, `severity_impact_mismatch`) + a tier (`strong`/`adequate`/`weak`) — the free, instant, deterministic counterpart to the paid LLM-as-judge. Wired in `main._run_analysis` (no `ceo_agent.py` edit); report gains `finding_quality` summary, and both finding views carry per-finding flags for the dashboard to badge. 3 new tests (61 total). The LLM self-reflection variant and structured-outputs (dropping the 2nd parse call — a ~40% call reduction) are the remaining agent-architecture rungs, but both change live API behaviour and must be validated on a real run before activation.

**Build 5 — Simulator bundle optimiser: SHIPPED.** `simulation.optimize_actions(report, scenario, max_actions, objective)` answers "which combination should I pursue first?" Because actions interact (margin + growth compound through `_project`), greedy ranking can be wrong — so it evaluates every bundle jointly via `apply_actions`; with ≤6 candidate actions this exhaustive search is **provably optimal** and cheap, and stays fully deterministic. Returns the best bundle (ties prefer the smaller bundle = less effort for the same lift), the top-5 bundles, and a marginal-lift curve (best achievable at each budget size). Objectives: `imara` / `profit` / `cash`. Exposed at `GET /api/report/{id}/optimize`. 4 new tests incl. a brute-force optimality check (65 total). Frontend "recommended bundle" panel is the surfacing follow-up.
