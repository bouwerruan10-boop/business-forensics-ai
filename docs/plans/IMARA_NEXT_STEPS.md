# Imara — Next Steps (research cycle conclusion)
*Honest framing: Imara has out-built its validation. After 6 deep feature workstreams it is feature-rich and internally validated (deterministic engine 100%, judge 100%), but pre-traction, uncalibrated, and not yet pressure-tested on real client runs. The research says the next leverage is NOT a 7th feature — it is **dependability + real-world validation**.*

## Research synthesis (two rounds)
- **Reliability.** Long-running LLM pipelines need **error classification** (a 429 ≠ a context overflow ≠ a quota outage), **idempotency**, **state across retries/restarts**, and **durable execution / checkpointing** — save progress at boundaries and resume from the last checkpoint, returning stored results for already-completed LLM calls so you don't re-pay tokens ([Inngest](https://www.inngest.com/blog/durable-execution-key-to-harnessing-ai-agents), [Diagrid](https://www.diagrid.io/blog/checkpoints-are-not-durable-execution-why-langgraph-crewai-google-adk-and-others-fall-short-for-production-agent-workflows)). Use SQLite/Postgres for production state, never in-memory ([LangChain](https://docs.langchain.com/oss/python/langgraph/durable-execution)).
- **Online evaluation.** Offline evals catch known regressions; **online evals** catch what offline misses — drift, edge cases, provider-side model changes — by reusing **deterministic checks on 100% of traffic** (instant malformed-output spike = early drift signal) + a **background judge on ~5% of runs** off the critical path ([TrueFoundry](https://www.truefoundry.com/blog/online-llm-evaluation-gateway), [VentureBeat](https://venturebeat.com/infrastructure/monitoring-llm-behavior-drift-retries-and-refusal-patterns)).
- **Traction.** The #1 reason AI products fail is poor **workflow fit / no tangible outcome**; **design partners** (5–12, time-bound, real workflows, weekly cadence, a hard "go paid or not" deadline) are the pre-launch edge that tells you what to build first and confirms willingness to pay ([Bessemer](https://www.bvp.com/atlas/design-partners-the-pre-launch-edge-most-ai-founders-ignore), [teamvoy](https://teamvoy.com/blog/why-most-ai-pilots-in-fintech-never-reach-production/)).

## Questions → answers
- **Q1 How fragile is Imara's pipeline?** Quite: a ~11-min, ~40-call FastAPI **background task** with no queue, no retries, no resume (it only marks a crashed run "interrupted"). A single transient provider blip can waste a whole run. **Answer:** add lightweight **checkpointing to the existing (now-persistent) SQLite** + error-classified retries — not heavy Temporal.
- **Q2 Can Imara see quality on real runs?** No — it has offline evals but no **online** eval. **Answer:** Imara already computes the right per-run signals (faithfulness conflicts, finding_quality strong/weak %, document coverage, extraction source, llm cost, macro resilience). It just needs to **aggregate them across the persisted analyses into a fleet quality dashboard + drift alerts** — cheap, because the signals already exist.
- **Q3 Is the real next step a feature at all?** Partly not. The highest-leverage move is a **design-partner pilot** that generates (a) the **real outcome data** that unblocks Score calibration (Layer 3) and (b) workflow-fit feedback. That is Ruan's to drive.

## Recommended next steps (prioritized)
1. **★ Online quality observability (buildable now, highest technical value).** Aggregate the signals Imara already produces across stored analyses into an **admin "Fleet Quality" dashboard + drift alerts**: faithfulness-conflict rate, % strong vs soft findings, avg cost/run, extraction-source mix, macro-resilience distribution, and an alert when any drifts vs a rolling baseline. This is the "online eval" the field calls standard, and Imara is uniquely ready for it (signals + persistence already in place).
2. **Pipeline durability (reliability).** Checkpoint each phase/agent output to SQLite as it completes; on restart/failure **resume from the last checkpoint** (don't re-pay ~40 calls); add **error-classified, bounded retries with backoff** on transient LLM errors. Makes real client runs dependable.
3. **Design-partner motion (validation / GTM — Ruan-led).** Recruit 5–12 representative SMEs/lenders, time-boxed, weekly cadence, hard "go paid" deadline. Generates the outcome data for calibration + tells us what to build next. Everything above makes Imara *ready* for this; this is what makes Imara *real*.

Sequencing rationale: (1) and (2) are self-contained, deterministic, no new data needed, and directly de-risk real-world use; (3) is the business step they unlock. None is another speculative feature — together they move Imara from "validated demo" to "dependable, observed, adopted."
