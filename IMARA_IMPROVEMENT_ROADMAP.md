## The honest headline

Imara is now **engineering-mature**: feature-complete for its purpose, reliability-hardened, security-hardened, and clean (CI ratchet in place). Across this whole session the recurring finding has been the same — **the bottleneck is no longer features; it's EVIDENCE and DISTRIBUTION.** The Imara Score is a well-built heuristic that has never been calibrated against real funding/repayment outcomes. Until it is, more agents and panels add surface area, not value.

So this roadmap is ordered by *what unblocks the business*, not by what's fun to build. Tier 0 is the real work and it's mostly yours to drive; Tiers 1–3 are what I can build, smallest-risk-highest-value first.

Your call stands; the plan just shouldn't pretend the contradiction isn't there.

## Tier 2 — Durability / scale (a concrete playbook; build at the volume inflection, not before)

Scaling is what a software engineer adds once a vibe-built product gets traction, and the failure modes are predictable. Honest stance for Imara: it is at **operator scale (~1–10 effective users)** today, so most of this is **premature now** — but the *plan* is cheap to write and expensive to retrofit under fire, so it lives here with Imara's real ceilings named.

**Imara's actual scaling ceilings (not the generic ones):**
- **SQLite write-lock is the first ceiling — not threads.** SQLite uses file-level locking → "database is locked" when analyses write concurrently (Postgres/MVCC fixes it). `database.py` already serialises writes with a lock; fine at operator scale, bites around 10–100 concurrent users.
- **Single backend instance** (one Railway container) → no horizontal scale / load balancing yet.
- **In-process background tasks** (no durable queue) → a restart kills in-flight runs (flagged, not resumed).
- **No DB backups** → the one do-now item (Tier 1.5): losing the volume loses all client data + the audit chain.
- **NOT a ceiling — don't "fix" it:** the "single-thread Python" trap does **not** apply here. The agents are **I/O-bound** (waiting on the Claude API) and Python **releases the GIL during network I/O**, so the existing ThreadPool wave fan-out is the *correct* design for this workload. Rewriting to multiprocessing would add complexity for no gain.

**Per-tier playbook (build a row when you cross into it):**

| Users | Dominant bottleneck | What to do |
|---|---|---|
| ~1–10 (now) | none structural | do now: DB backups (1.5) + error tracking (1.4); SQLite **WAL mode** is a cheap concurrency stopgap |
| ~10–100 | SQLite write-lock; restart loses jobs | durable **job queue** (arq/RQ/Celery) + concurrency cap; **migrate to Postgres**; keep the `/api/health` check |
| ~100–1k | single instance | multiple workers behind a **load balancer**; Postgres connection pooling; response/result caching |
| ~1k–10k+ | DB + compute | autoscaling workers; Postgres **read replicas**; CDN (already on Vercel); queue scaling; per-tenant isolation (seams already shipped) |
